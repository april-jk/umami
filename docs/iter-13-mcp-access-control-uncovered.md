# 迭代 13: MCP 安装识别与访问控制 - 未覆盖测试说明

## 未覆盖场景

- Railway 生产环境执行 `29_mcp_client_access` 数据库迁移需要真实 PostgreSQL 权限和已备份的生产数据，不能由单元测试覆盖。
- MCP 客户端安装后的真实 API 请求、UA 采集、IP 日哈希和数据库访问记录属于跨服务链路；单元测试已经覆盖请求解析与写入参数，但不能证明生产网络、反向代理和数据库连接均正常。
- API key 最后使用时间与 MCP 访问记录均采用非阻塞写入。写入失败不应影响授权请求；真实数据库暂时故障下的可观测性与告警需要在部署环境验证。
- MCP 客户端版本兼容性与更新提示依赖已发布的 npm 包版本，不能用本仓库单元测试替代实际升级验证。

## 手动测试步骤

1. 在预生产数据库中以事务执行迁移，确认 `api_key.client_type` 和 `mcp_client_access` 表、索引均已创建；先回退应用版本，再执行同目录的 `rollback.sql` 或恢复演练确认可恢复。
2. 使用新建的 MCP API key 调用一个受保护的 API，并携带 `x-amami-mcp-client` 与正常 User-Agent；确认请求成功，且访问表只保存截断后的 UA、客户端版本和 IP 日哈希。
3. 使用普通 API key 重复调用，确认不会创建 MCP 安装访问记录；伪造 MCP 头也不得改变该结论。
4. 临时使访问记录写入失败，确认 API 仍完成授权并在服务日志中留下失败信息。
5. 使用已发布与过期的 MCP 包各执行一次客户端策略检查，确认兼容状态与升级提示符合产品策略。

## 建议补充的测试用例

- 在 CI 的临时 PostgreSQL 中执行迁移、写入访问记录并验证索引和数据最小化字段。
- 增加 MCP CLI 到服务端的端到端测试，覆盖升级提示和真实 HTTP 头传递。
- 为 Railway 部署后检查建立只读健康探针：验证新表存在、写入权限可用且不输出原始 IP。
