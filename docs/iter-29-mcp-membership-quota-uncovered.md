# 迭代 29：会员 MCP 日/月配额强制执行未覆盖说明

## 已完成的自动化覆盖

- Free、Starter 的 UTC 日配额，以及 Pro、Team 的 UTC 月配额选择。
- Enterprise 无限配额、零额度和额度边界调用。
- PostgreSQL 计数器的事务 upsert、条件递增和达到边界后的拒绝结果。
- MCP API Key 认证前的额度预留、429 `plan-limit` 响应上下文和拒绝访问记录。
- 管理员套餐配置、单租户 MCP 覆盖值、普通会员页用量条和升级页周期标签。
- 历史会员配置缺少 `mcpCallsPerMonth` 字段时按当前代码默认值回填。
- 52 个语言包的 MCP 通用、日调用和月调用标签。

## 无法由单元测试完整覆盖的场景

### 1. Railway PostgreSQL migration 与真实并发

原因：单元测试使用 Prisma mock，无法证明 production Postgres 已执行 migration 32，也无法模拟多实例同时抢占最后一个额度的真实事务行为。

手动步骤：

1. 部署后确认启动日志显示 migration 32 已应用且没有待执行 migration。
2. 在预生产为测试租户设置一个很小的日额度，例如 2。
3. 使用同一租户下两个 MCP API Key 并发发起 5 个受保护 API 请求，确认成功数不超过 2，其余响应为 429。
4. 查询管理员用户会员页，确认当前用量、剩余量和拒绝记录一致。

### 2. 真实 MCP 客户端链路

原因：测试没有运行真实 `amami-analytics-mcp` 客户端、PKCE 授权、API Key 保存和多个工具调用。

手动步骤：

1. 使用 Free、Starter、Pro 测试租户分别完成 MCP 安装授权。
2. 连续调用工具直到达到边界，确认客户端收到 `mcp-calls-limit-reached`，且 `/api/mcp/client-policy` 不消耗 MCP 配额。
3. 切换套餐或兑换更高等级激活码，确认新周期/新额度立即生效。

### 3. Redis 故障降级

原因：本实现使用数据库计数器作为权威状态，仍需验证 Railway Redis 不可用时 MCP 认证不会绕过额度，也不会因可选缓存故障丢失计数。

手动检查：

- [ ] 暂停 Redis 后重复执行日/月边界调用，确认限额仍由 Postgres 执行。
- [ ] 恢复 Redis 后确认普通 MCP 请求和访问记录继续正常。

### 4. 多语言和响应式浏览器检查

原因：单元测试只能验证消息键与渲染文本，不能覆盖真实浏览器中的长德语、阿拉伯语 RTL、移动端横向布局和 assistive technology 读取顺序。

检查清单：

- [ ] 在 `zh-CN`、`en-US`、`de-DE`、`ar-SA`、`ja-JP` 切换会员页和升级页。
- [ ] 在 390px 和 1280px 视口确认 MCP 用量条、周期标签和升级入口不重叠。
- [ ] 确认管理员覆盖编辑器的日/月控件拥有可读 label，键盘可以完成保存。

## 风险与回滚

- migration 32 只新增计数器表，不修改既有业务表；回滚应用代码后该表可保留，不影响既有 MCP 访问记录。
- 如需禁用执行，可移除 `MEMBERSHIP_ENABLED`/`CLOUD_MODE` 的会员执行开关，或在管理员配置中将对应 MCP 周期设为无限；不应直接删除计数器数据。
- 当前配额按租户共享，所有该租户 API Key 共用额度；这是防止通过创建多个 Key 绕过会员限制的有意决策。
