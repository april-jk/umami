# 迭代 14: MCP 调用内更新提示 - 未覆盖测试说明

## 未覆盖场景

- 不同 MCP host 对一个工具结果中多个 text content block 的上下文呈现方式属于宿主客户端行为，不能由 MCP 服务的单元测试覆盖。
- 更新策略请求设置为 1 秒超时、无重试且失败静默；真实网络拥塞、DNS 故障和宿主进程取消信号只能在实际 MCP transport 中验证。
- “每个服务实例只提示一次”以 MCP 服务进程内存为边界。客户端重启、HTTP serverless 冷启动或横向扩容后重新提示属于预期行为，需在部署环境确认其频率符合产品预期。

## 手动测试步骤

1. 使用发布版 MCP 包连接一个支持多次 tool call 的 MCP host，并让服务端策略返回高于当前版本的 `latestVersion`。
2. 调用任意正常读取工具，确认业务结果完整，末尾仅出现一次简短更新提示和文档链接。
3. 在同一连接内继续调用至少三个工具，确认不再出现更新提示，也不会重复请求客户端策略接口。
4. 让 `/mcp/client-policy` 返回超时或 5xx，确认工具的业务结果仍正常返回，且不显示更新提示。
5. 显式调用 `get_client_update_status`，确认仍能获得完整结构化兼容策略，不会附加重复通知。

## 建议补充的测试用例

- 在 Claude Desktop、Codex 和一个 HTTP MCP host 中分别运行端到端脚本，记录多个 text content block 的呈现及上下文消耗。
- 在预生产环境注入 1 秒以上的策略接口延迟，验证业务工具不会重试更新检查且请求在超时后静默完成。
- 为常驻 stdio 与 serverless transport 分别添加运行时指标，观察更新提示投递率和策略接口失败率。
